# رویکرد TDD

این پروژه از ابتدا با تست‌های قرارداد اصلی ساخته شده است. قاعده توسعه:

1. برای هر رفتار جدید اول تست backend یا frontend نوشته شود.
2. اگر رفتار به AI مربوط است، اول با `FakeAiProvider` تست deterministic نوشته شود.
3. تست live provider فقط با marker `ai_live` اجرا شود تا توسعه محلی هزینه API نداشته باشد.
4. بعد از سبز شدن تست‌ها، UI یا refactor انجام شود.

## سناریوهای پایه

- کاربر در صفحه اول مستقیم از عکس محصولات شروع کند.
- واژه‌های داخلی مثل «بچ» و «انتخاب فروشنده» در UI اصلی دیده نشوند.
- هر فروشنده محلی همچنان دیتای جدا داشته باشد.
- bootstrap فرانت نباید اولین seller دیتابیس را به کاربر جدید نسبت بدهد؛ seller فعلی فقط از `localStorage` همان مرورگر یا `seller_id` معتبر callback باسلام می‌آید و `GET /sellers` نباید لیست عمومی sellerها را expose کند.
- مسیرهای باسلام و ترب نباید batch، عکس، ویس، item، job یا publish state مشترک داشته باشند. با انتخاب هر مسیر، workspace تازه برای همان مسیر ساخته می‌شود.
- ترتیب عکس‌ها بعد از upload، refresh و پردازش ثابت بماند.
- اگر AI fail شد، batch و فایل‌ها حذف نشوند.
- در e2e مرورگری، اگر پردازش AI fail شد، عکس‌ها باید روی صفحه بمانند، دکمه retry کار کند، و متن خام انگلیسی/فنی مثل خطای provider به کاربر نمایش داده نشود.
- edit محصول باید `edited_by_user=true` کند.
- CSV باید ستون‌های لازم برای بررسی انسانی و import بعدی را داشته باشد.
- اتصال باسلام باید بدون مصرف API واقعی با fake client تست شود.
- callback باسلام باید فقط با `state` معتبر اتصال بسازد.
- callback باسلام باید هر seller را فقط به غرفه خودش وصل کند؛ اتصال یک کاربر نباید باعث شود کاربران دیگر به همان غرفه وصل شوند.
- اگر OAuth باسلام در لوکال تنظیم نشده باشد، endpoint ساخت لینک باید `configured=false` بدهد، نه 503.
- فروشنده باید بتواند بعد از ساخت و تکمیل لیست هم از همان CTA پایین صفحه غرفه باسلام را وصل کند؛ لینک OAuth باید با seller همان مرورگر ساخته شود.
- اگر فروشنده بعد از ویرایش لیست از دکمه اتصال بالای پنل باسلام استفاده کرد، فیلدهای ویرایش‌شده باید قبل از رفتن به OAuth ذخیره شوند.
- اگر فروشنده بعد از تکمیل لیست برای اتصال باسلام به OAuth رفت و برگشت، باید به همان مسیر و همان batch برگردد و فیلدهای پرشده از بین نروند.
- اگر فروشنده قبل از ساخت لیست یا قبل از upload جدی غرفه باسلام را وصل کرد و از OAuth برگشت، باید همچنان در مسیر باسلام و صفحه upload بماند، نه اینکه به انتخاب مسیر برگردد.
- اگر localStorage به هر دلیل به batch فروشنده دیگری اشاره کرد، بعد از OAuth نباید آن batch restore شود؛ workspace تازه برای seller فعلی ساخته می‌شود.
- اگر OAuth به اشتباه با `code/state` مستقیم به frontend برگشت، frontend باید همان query را به callback backend پاس بدهد تا کاربر از مسیر باسلام بیرون نیفتد.
- عکس‌های خیلی بزرگ باید قبل از upload در مرورگر کوچک شوند تا upload برای فروشنده کند و فرسایشی نشود؛ آماده‌سازی عکس‌ها با concurrency محدود انجام می‌شود، اما آرایه نهایی باید ترتیب انتخاب کاربر را حفظ کند تا شماره عکس‌ها جابه‌جا نشوند. اگر resize ممکن نبود، فایل اصلی با fallback امن ارسال می‌شود.
- هنگام upload کند، UI باید همان لحظه loading مینیمال «در حال آماده‌سازی عکس‌ها» نشان بدهد و کاربر را در سکوت نگه ندارد.
- اگر ارتباط frontend با backend قطع شد، UI باید پیام فارسی ساده نشان بدهد و متن خامی مثل `Failed to fetch` دیده نشود.
- جستجو و انتخاب دسته‌بندی باسلام باید با fake category tree تست شود.
- publish باسلام باید قبل از ثبت، محصولات ویرایش‌شده را ذخیره کند و نتیجه هر محصول را جدا نگه دارد.
- ذخیره draft محصول در frontend نباید metadata پلتفرم را از بین ببرد؛ اگر پاسخ PATCH دسته‌بندی باسلام یا عکس‌ها را برنگرداند، UI باید مقدار قبلی همان محصول را حفظ کند تا کاربر بعد از پر کردن فیلدها ناگهان با خطای دسته‌بندی ناقص روبه‌رو نشود.
- publish موفق باسلام باید در UI و e2e مرورگری پیام قابل فهم «محصول‌ها در باسلام ثبت شدند» و تعداد محصول ثبت‌شده را نشان دهد.
- publish باسلام وقتی اطلاعات لازم ناقص است باید قبل از upload عکس و create محصول متوقف شود.
- اگر بخشی از publish شکست خورد، محصولات موفق نباید گم شوند و job باید وضعیت قابل فهم داشته باشد.
- خطاهای فنی/انگلیسی باسلام در publish نباید خام به کاربر نشان داده شوند؛ UI باید پیام فارسی قابل اقدام نشان دهد.
- دلیل fail شدن publish باید هم در پنل نتیجه بالای لیست و هم در dock پایین صفحه دیده شود؛ چون کاربر بعد از ویرایش معمولا پایین صفحه است و نباید مجبور شود برای فهمیدن خطا اسکرول کند.
- اگر باسلام دسته‌بندی پیشنهادی AI را قبول نکرد، پیام خطا نباید بگوید «انتخاب نشده»؛ باید توضیح دهد دسته پیشنهادی پذیرفته نشده و کاربر از دکمه «تغییر» دسته نزدیک‌تر انتخاب کند.
- اگر دسته‌بندی auto در publish توسط باسلام رد شد، backend باید بدون دخالت فروشنده چند دسته نزدیک بعدی از همان shortlist را retry کند؛ اگر یکی موفق شد، `BatchItemPlatformData` همان دسته موفق را ذخیره کند.
- اگر کاربر بدون ویس لیست ساخت و بعد اطلاعات ناقص بود، بتواند در همان پیام پایین صفحه صدا ضبط کند و همان batch دوباره پردازش شود.
- اگر کاربر قبل از ساخت لیست ویس ضبط کرد، audio asset باید کنار عکس‌های همان batch ذخیره شود و همان batch با صدا پردازش شود.
- فرمت ضبط صدا نباید در frontend به `webm` جعل شود؛ MIME و پسوند فایل باید از خروجی واقعی `MediaRecorder` بیاید تا Safari/MP4 و Chrome/WebM هر دو کار کنند.
- فایل صوتی خالی نباید upload شود و باید پیام فارسی «صدایی ضبط نشد» نمایش داده شود.
- upload صدا نباید spinner دکمه یا شبکه عکس را فعال کند؛ عکس و صدا state بارگذاری مستقل دارند.
- timeout، قطع ارتباط، rate limit، خطای 5xx، JSON نامعتبر و خروجی خالی AI باید حداکثر سه تلاش کنترل‌شده داشته باشند.
- شکست نهایی AI باید فایل‌های خام را نگه دارد، پیام امن فارسی بدهد و `code/stage/attempts/exception_type` را در `Batch.ai_metadata.last_processing_failure` ثبت کند.
- log شکست pipeline باید `job_id`, `batch_id`, `stage`, `code`, `attempts` و traceback داشته باشد؛ متن فنی نباید در `ProcessingJob.error` عمومی قرار بگیرد.
- CTA پیام اطلاعات ناقص باید متن روشن داشته باشد؛ «تکمیل فیلدها» به اولین کارت ناقص می‌رود.
- مسیر مرورگری باسلام و ترب باید جدا تست شود؛ تغییر مسیر نباید عکس‌ها یا batch قبلی را وارد مسیر جدید کند.
- مسیر مرورگری ترب باید تا ثبت درخواست پیش برود و ثابت کند فیلدهای اختصاصی باسلام یا حدس دسته‌بندی باسلام در آن مسیر فعال نمی‌شوند.
- مسیر ترب هم باید اجازه ضبط صدا قبل از ساخت لیست بدهد، اما نباید endpoint حدس دسته‌بندی باسلام را صدا بزند.
- UI اصلی در e2e دسکتاپ و موبایل نباید overflow افقی داشته باشد و پنل‌ها/کارت‌ها در موبایل باید داخل gutter امن صفحه بمانند.
- باکس «زمان آماده‌سازی همه محصولات» در موبایل نباید input و دکمه اعمال را روی هم بیندازد و دکمه بستن آن باید حداقل ۳۴ پیکسل سطح لمس داشته باشد.
